## 大局观：DevOps 与云计算

当涉及到云计算及其提供的内容时，它与 DevOps 的理念和流程非常契合。我们可以将云计算视为带来技术和服务，而 DevOps 正如我们多次提到的，是关于流程和流程改进的。

但开始这段云计算学习之旅是陡峭的，确保你了解和理解所有要素或选择最佳服务以获得合适的价格点是令人困惑的。

![](Images/Day28_Cloud1.png)

公共云是否需要 DevOps 思维？我的答案是：不需要，但要真正利用云计算并可能避免那么多人遇到的高额云账单，那么将云计算和 DevOps 一起思考是很重要的。

如果我们从 40,000 英尺的高度看待公共云的含义，它是关于将一些责任转移给托管服务，以使你和你团队能够专注于更重要的方面，这些方面应该是指应用程序和最终用户。毕竟，公共云只是别人的电脑。

![](Images/Day28_Cloud2.png)

在这第一部分中，我想更深入地了解公共云是什么以及构成公共云整体的那些构建块。

### SaaS

第一个要涵盖的领域是软件即服务（Software as a Service），这项服务几乎消除了你可能曾经在本地运行的某个服务的所有管理开销。让我们想想用于我们电子邮件的 Microsoft Exchange，这过去是一台物理盒子，放在你的数据中心或者可能放在楼梯下的橱柜里。你需要喂养和照顾那台服务器。我的意思是，你需要保持它更新，你要负责购买服务器硬件，很可能还要安装操作系统、安装所需的应用程序，然后保持它的补丁更新，如果出了什么问题，你必须进行故障排除并让事情恢复运行。

哦，你还必须确保你在备份你的数据，尽管这在很大程度上也不会因为 SaaS 而改变。

SaaS 所做的是，特别是 Microsoft 365，因为我提到了 Exchange，它消除了那部分管理开销，他们提供了一项服务，通过邮件的方式提供你的 Exchange 功能，但也提供许多其他生产力（Office 365）和存储选项（OneDrive），总体上为最终用户提供了出色的体验。

其他 SaaS 应用被广泛采用，例如 Salesforce、SAP、Oracle、Google、Apple。都在消除必须管理更多技术栈的负担。

我相信 DevOps 和基于 SaaS 的应用程序之间有一些故事，但我很难找到它们可能是什么。我知道 Azure DevOps 与 Microsoft 365 有一些很好的集成，我可能会研究一下并回报。

![](Images/Day28_Cloud3.png)

### 公共云

接下来我们有公共云，大多数人会对此有几种不同的看法，有些人会认为这只是超大规模云服务商，例如 Microsoft Azure、Google Cloud Platform 和 AWS。

![](Images/Day28_Cloud4.png)

还有些人会将公共云视为更广泛的 offerings，包括那些超大规模云服务商，还有全世界成千上万的 MSP。对于这篇文章，我们将把公共云视为包括超大规模云服务商和 MSP，尽管稍后我们会专门深入探讨一个或多个超大规模云服务商，以获得基础知识。

![](Images/Day28_Cloud5.png)
*成千上万的公司可以列在这里，我仅仅是从我合作过和了解过的本地、区域、电信和全球品牌中挑选。*

我们在 SaaS 部分提到，云消除了管理或管理系统的责任或负担。在 SaaS 中，我们看到很多抽象层被移除了，即物理系统、网络、存储、操作系统，甚至在某种程度上还包括应用程序。当涉及到云时，根据你的需求，有各种级别的抽象我们可以移除或保留。

我们已经提到了 SaaS，但关于公共云至少还有两项要提及。

基础设施即服务（Infrastructure as a Service）- 你可以将这一层看作虚拟机，但与本地不同，在本地你必须照看物理层，在云中情况并非如此，物理层是云提供商的责任，你将管理和维护操作系统、数据以及你希望运行的应用程序。

平台即服务（Platform as a Service）- 这继续移除了各层的责任，这实际上是关于你掌控数据和应用程序，但不必担心底层硬件或操作系统。

还有许多其他的 *aaS offerings，但这两个是基础。你可能会看到围绕 StaaS（Storage as a service，存储即服务）的产品，它为你提供存储层，但不必担心下面的硬件。或者你可能听说过 CaaS（Containers as a Service，容器即服务），我们稍后会讲到，接下来 7 天我们要了解的另一个 aaS 是 FaaS（Functions as a Service，函数即服务），也许你不希望一直运行一个系统，你只想让一个函数在需要时执行。

公共云可以通过许多方式提供你希望放弃并付费的控制抽象层。

![](Images/Day28_Cloud6.png)

### 私有云

拥有自己的数据中心并不是过去的事情，我认为这在很多发现 OPEX 模式难以管理以及仅使用公共云的技能集的公司中得到了复兴。

这里需要注意的是，公共云现在可能将是你的责任，而且它将在你的本地。

在这个领域，我们有一些有趣的事情正在发生，不仅有 VMware，它主导了虚拟化时代和本地基础设施环境。我们还有超大规模云服务商提供其公共云的本地版本。

![](Images/Day28_Cloud7.png)

### 混合云

紧随公共云和私有云的提及之后，我们还可以跨越这两个环境，以在两者之间提供灵活性，也许可以利用公共云中可用的服务，然后还可以利用本地的功能和特性，或者可能是规定你必须在本地存储数据的法规。

![](Images/Day28_Cloud8.png)

将所有这些放在一起，我们在存储和运行工作负载的位置有很多选择。
