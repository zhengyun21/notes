## 计划 > 编码 > 构建 > 测试 > 发布 > 部署 > 运维 > 监控 >

今天我们将关注从始至终的各个步骤，以及DevOps世界中应用程序的持续循环。

![DevOps](Images/Day5_DevOps8.png)

### 计划：

一切都始于计划过程，开发团队在此聚在一起，弄清楚他们将在下一个冲刺（sprint）中推出哪些类型的功能和bug修复。作为DevOps工程师，这是你参与其中的机会，了解哪些事情将要发生并且你需要参与其中，同时影响他们的决策或方向，并帮助他们使用你已经构建的基础设施，或者在他们的方向不正确时引导他们走向更好的方案。这里需要指出的一个关键点是，开发人员或软件工程团队作为DevOps工程师的客户，这是你与客户合作的机会，在他们走上错误道路之前。

### 编码：

一旦计划会议结束，他们将开始编写代码。你可能参与很多，也可能参与不多。你可能参与的一个地方是，当他们编写代码时，你可以帮助他们更好地理解基础设施，让他们知道有哪些可用的服务以及如何最好地与这些服务通信。他们会这样做，一旦完成，他们就会将代码合并到仓库中。

### 构建：

这是我们将启动第一个自动化流程的地方，因为我们将获取他们的代码并进行构建。根据他们使用的语言，可能是转译、编译，或者可能是从该代码创建Docker镜像。无论如何，我们将通过我们的CI/CD管道来完成这个过程。

## 测试：

一旦我们构建了它，我们就要对其进行一些测试。现在，开发团队通常会编写测试，你可能对编写什么测试有一些意见，但我们需要运行这些测试。测试是我们尽量减少将问题引入生产环境的一种方式，它不能保证这一点，但我们希望尽可能地接近保证：一是不引入新的bug，二是不破坏过去正常工作的功能。

## 发布：

一旦这些测试通过，我们将进行发布流程。同样，根据你正在开发的应用程序类型，这可能不是一个步骤。你知道代码可能只存在于GitHub仓库或Git仓库或它所在的任何地方，但它可能是将你编译好的代码或构建好的Docker镜像放入注册表或仓库的过程，以便你的生产服务器可以在部署过程中访问它。

## 部署：

这是我们接下来要做的事情，因为部署就像整个过程的最终目标，因为部署是我们将代码投入生产的时候，只有到那时，我们的业务才能真正意识到你和软件工程团队迄今为止在这个产品上投入的所有时间、努力和辛勤工作的价值。

## 运维：

一旦部署完成，我们就要对其进行运维。运维可能涉及一些事情，比如你开始接到客户的电话，他们都抱怨网站运行缓慢或他们的应用程序运行缓慢，所以你需要找出原因，然后可能构建自动扩展来处理高峰期增加可用服务器数量，并在非高峰期减少服务器数量。无论如何，这些都是运维类型的指标。你做的另一件运维事情是包含一个从生产环境回到运维团队的反馈循环，让你知道生产环境中发生的关键事件，比如部署。关于部署，后退一步说，这可能实现自动化也可能不实现自动化，取决于你的环境。目标是尽可能实现自动化。在某些环境中，你可能需要执行几个步骤才能准备好实现自动化，但理想情况下，你希望将自动部署作为自动化过程的一部分。但如果你这样做，在你的运维步骤中包含某种类型的通知可能是个好主意，这样你的运维团队就知道已经发生了部署。

## 监控：

以上所有部分都导向最后一步，因为你需要监控，特别是围绕运维问题、自动扩展、故障排除。就像如果你没有任何监控来告诉你有问题，你就不知道有问题一样。所以你可能会为之构建监控的一些内容包括内存利用率、CPU利用率、磁盘空间、API端点、响应时间、该端点响应的速度，以及其中很大一部分是日志。日志使开发人员能够看到正在发生的事情，而无需访问生产系统。

##  rinse and repeat（重复循环）：

一旦这些就位，你就直接回到最初的计划阶段，再次经历整个过程。

## 持续：

许多工具帮助我们实现上述持续流程，所有这些代码以及完全自动化的最终目标，云基础设施或任何环境通常被描述为持续集成/持续交付/持续部署，或简称"CI/CD"。我们将在90天后面的时间里用整整一周的时间来学习CI/CD，并附有一些示例和演练来掌握基础知识。

### 持续交付：

Continuous Delivery = Plan > Code > Build > Test

### 持续集成：

这实际上是上述持续交付阶段的结果加上发布阶段的结果。无论失败还是成功都是如此，但这会反馈到持续交付中，或转移到持续部署。

Continuous Integration = Plan > Code > Build > Test > Release

### 持续部署：

如果你从持续集成中获得成功的发布，那么就转移到持续部署，这将引入以下阶段：

CI Release is Success = Continuous Deployment = Deploy > Operate > Monitor

你可以将上述三个持续概念视为DevOps生命周期阶段的简单集合。
