15. map的底层原理
	1. HashMap：jdk1.7为数组和链表，jdk1.8之后为数组、链表和红黑树（O(Log n)）
	2. LinkedHashMap：在 HashMap 的数组+链表/红黑树结构之上，增加了一条**双向链表**，串联起所有的 Entry。
	3. TreeMap：基于红黑树red-black tree，有序（O(log n)）
	4. ConcurrentHashMap：**多线程**首选，jdk1.7为segment锁，jdk1.8之后为CAS+Synchronized锁
16. map的put过程以及扩容机制，什么时候转成链表，什么时候转成红黑树
	1. 先判断table是否为空，空则初始化容量为16的map
	2. 先计算key的HashCode和索引值，然后定位到数组索引位置进行比较，如果没有元素或者相同的话就直接覆盖掉value，如果不同则进行尾插法插入
	3. 当数组长度 >= 64 和 链表长度 > 8 时，进行红黑树的转换 
	4. 当容量超过负载因子0.75的阈值，则触发扩容
17. concurrentHashmap在jdk1.8前和后有什么区别和不同
	1. jdk1.8前是用segment锁保证并发安全，理论上运行16个线程同时操作
	2. jdk1.8之后是用CAS+Synchronized锁保证并发安全，当该位置为空，则CAS进行插入，无需加锁；如果不为空（存在哈希冲突），就用Synchronized锁住头节点，防止其他线程操作
18. synchronized和lock锁的区别
	1. synchronized是Java关键字，在JVM层面实现加锁和解锁；Lock是一个接口，在代码层面实现加锁和解锁。
	2. synchronized可以用在代码块上、方法上；Lock只能写在代码里。
	3. synchronized在代码执行完或出现异常时自动释放锁；Lock不会自动释放锁，需要在finally中显示释放锁。
	4. synchronized会导致线程拿不到锁一直等待；Lock可以设置获取锁失败的超时时间。
	5. synchronized无法得知是否获取锁成功；Lock则可以通过tryLock得知加锁是否成功。
	6. synchronized锁可重入、不可中断、非公平；Lock锁可重入、可中断、可公平/不公平，并可以细分读写锁以提高效率。
19. synchronized锁的升级机制
	1. 无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁
	2. 当一个无锁的线程尝试获取锁，就获得了偏向锁
	3. 偏向锁出现了锁的竞争，如果自旋一次后锁还在，则转为轻量级锁，否则则释放锁给其他线程
	4. 如果锁竞争激烈，则转为重量级锁
20. 自旋锁（spinlock）是什么
	1. 线程在获取锁失败时，不会立即阻塞，而是会通过循环检测锁的状态来再次尝试获取锁

22. **JMM**内存模型、**JVM**内存结构
	1.  JMM内存模型指的是Java语言规范定义的**多线程环境下内存访问的规则和约束**，核心目标是**解决并发编程中的内存可见性、原子性和有序性问题**。
		1. 主内存：**所有线程共享的内存区域，存储变量的实际值**
		2. 工作内存：**每个线程私有的内存区域，存储主内存中变量的副本**
		3. 三大特征：**可见性、原子性、有序性**
		4. 关键工具：**volatile、final、synchronized**
	2. 是JVM在运行Java程序时对**物理内存的划分方式**。

24. 最大线程数和核心线程数的区别
	1.  核心线程数指的是线程池中保持的最小线程数，即使是空闲的
	2.  最大线程数指的是当任务队列已满，核心线程也在处理时，线程池会创建新的线程，知道达到最大线程数。
25. 线程池的执行流程
	1. 线程池在获取任务后会先访问核心线程，如果空闲的话就直接调用线程进行任务处理；如果核心线程都被占用，就进入任务队列进行等待；如果任务队列也满了，就开辟新的线程进行任务处理；如果所有线程的数量已经到达了最大线程数，就触发拒绝策略。

			2. MVCC指的是一条记录会有多个版本，每次修改记录时会记录修改前的版本，多个版本串联起来形成一条版本链

37. RDB和AOF了解吗
	1. 都是Redis持久化的方案
	2. RDB：类似快照，通过save或者bgsave指令来存储redis当中某一时刻的内容
		1. save和bgsave区别：
			1. save会阻塞所有客户端请求，等执行完RDB操作之后才会取消阻塞
			2. bgsave则是异步执行RDB操作，redis会创建一个线程来专门进行RDB操作。
		2. 优点：
			1. 速度快、内存占用少，适用于快速恢复且数据影响不大的场景
		3. 缺点：
			1. 影响性能
			2. 可能造成数据丢失，因为是定期快照存储方式
	3. AOF：记录每一次的写操作，通过 AppendOnly yes 开启
		1. 优点：
			1. 安全性高、可操作性强，可以对文件进行修改。适用于对数据安全性高的场景
		2. 缺点：
			1. 文件体积大
			2. 恢复速度慢
	4. 一般建议混合使用RDB和AOF。
38. Redission分布式锁原理
	1. 通过redis的setNX和DEL命令来创建分布式锁，setNX创建成功则返回1，失败返回0
	2. 看门狗机制：
		1. 通过一个定时任务（看门狗）来检查锁的状态：
			1. 默认锁的过期时间是30秒，每10秒检查一次，没过期则续约至30秒
			2. leaseTime：也可以对锁设置leaseTime过期时间
			3. waitTime：当waitTime > 0 时，线程会在指定时间内尝试获取锁。如果获取不到锁，tryLock会返回False。（前提是lock被其他线程拿了。）

48. JVM的垃圾回收机制
	1. 新生成的对象会先存到Eden区，Eden区满了会触发Minor GC，存活下来的对象会被移动到S0区，S0满了触发MinorGC，存活的移动到S1区；S1满了又触发MinorGC，然后又存到S0区。这样反反复复，直到达到15次时，存活的对象会被移动到老生代区。
	2. 老生代可能会触发Full GC，条件是老生代满了或者是手动调用system.gc，前提是触发了MinorGC
49. 垃圾回收方法有哪些
	1. 复制：把内存分为两块，然后使用其中的一块，当其中一块满了就复制到另一块，然后清除掉前一块
		1. 特点：
			1. 可用内存变小
			2. 不适合老生代
	2. 标记清除：标记不需要回收的对象，然后回收其他对象
		1. 特点：
			1. 效率低
			2. 产生不连续的内存碎片
	3. 标记整理：把存活的对象放到一起，然后清除掉端边界以外的内存
		1.  特点：

53. HotSpot 为什么要分为新生代和老年代？
	1. 比如在新生代中，每次收集都会有大量对象死去，所以可以选择“复制”算法，只需要付出少量对象的复制成本就可以完成每次垃圾收集。而老年代的对象存活几率是比较高的，而且没有额外的空间对它进行分配担保，所以我们必须选择“标记-清除”或“标记-整理”算法进行垃圾收集。

60. AQS（AbstractQueuedSynchronizer）

		2. **CLH队列**：线程获取锁失败时，线程会作为节点加入队列尾部，当state释放后（0），就会唤醒第一个节点

		1. 客户端发送FIN给服务器

		3. 服务器再次发送FIN给客户端
		4. 客户端接收后发送ACK给服务器，两者断开连接

65. cookie、session、token、JWT区别
	1. Http是无状态协议，服务器无法区分上一次请求与下一次请求是不是同一个客户端发送出来的，为了区别，就需要去维护一个状态，这个状态需要通过session或者cookie来实现.
	2. Cookie：是服务器发送给客户端浏览器并保存在本地的一段数据，它会在下一次请求服务器时携带上.
		1. Cookie不可跨域.
		2. 只支持字符串
		3. 有效期长
		4. 存储数据小
	3. Session：是基于Cookie实现的，Session存在服务器当中，SessionId存在本地.
		1. 当第一次请求时，服务器会返回内容+SessionId.
		2. 第二次请求时，客户端会携带SessionId一同发送给服务器，服务器验证Id后才会进行下一步操作.
			1. 比Cookie安全
			2. 支持任意数据类型
			3. 有效期短
			4. 存储数据大
	4. Token（令牌）：
		1. 组成：uid、time、sign等
		2. 验证方式：需要服务端查询数据库或者缓存来验证有效性，通常是有状态的
	5. JWT：
		1. 组成：header、payload、signature
		2. 验证：通过签名验证，无需查库直接判断是否被篡改，通常是无状态的 
			1. **token需要查库验证token 是否有效，而JWT不用查库或者少查库，直接在服务端进行校验，并且不用查库。**
