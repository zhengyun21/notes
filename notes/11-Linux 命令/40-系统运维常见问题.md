## 一、CPU 负载排查

### 1.1 核心命令：`top`

`top` 是 Linux 下最经典的实时性能监控工具，能一眼看清系统整体状态。

```bash
top
```

> **阅读技巧**：`top` 输出分为上下两部分——**上半部分看系统整体**，**下半部分看具体进程**。先判断整体是否异常，再定位具体进程。

---

### 1.2 上半部分：系统整体概况（前 5 行）

#### 第 1 行：系统运行时间与负载

| 字段 | 含义 | 关键判断 |
|------|------|----------|
| `top - 15:30:16` | 当前系统时间 | — |
| `up 23:57` | 系统已运行时间 | 越长说明越稳定 |
| `2 users` | 当前登录用户数 | 注意是否有异常用户 |
| `load average: 4.08, 3.56, 1.96` | **1/5/15 分钟平均负载** | **数值 > CPU 核数 = 过载** |

> **负载理解**：负载值代表"平均活跃进程数"。比如四核 CPU，负载长期 > 4 说明系统已严重过载，排队进程越来越多。

#### 第 2 行：进程状态统计

| 字段 | 含义 | 正常范围 |
|------|------|----------|
| `Tasks: 129 total` | 总进程数 | 视系统而定 |
| `5 running` | **正在运行/等待运行的进程** | 如果 CPU 不忙碌，这个值应该很小 |
| `124 sleeping` | 休眠进程 | 大多数进程应该在此状态 |
| `0 stopped` | 被暂停的进程 | 通常为 0 |
| `0 zombie` | **僵尸进程** | **理想为 0**，有值说明父进程未回收子进程 |

> **僵尸进程**：子进程已结束但父进程未调用 `wait()` 回收资源。少量无害，大量可能说明程序有 bug。

#### 第 3 行：CPU 状态（⚠️ 最核心的参数）

| 字段 | 全称 | 含义 | 异常信号 |
|------|------|------|----------|
| `59.0 us` | user | **用户空间占用 CPU 时间** | 过高说明应用程序在疯狂跑 |
| `40.2 sy` | system | **内核空间占用 CPU 时间** | 过高说明频繁调用系统接口（如 IO 操作） |
| `0.0 ni` | nice | 改变过优先级的进程占用 | 一般很少见 |
| `0.0 id` | idle | **空闲 CPU 百分比** | **0% 说明 CPU 完全被打满！** |
| `0.0 wa` | wait | **等待 I/O 的时间百分比** | 有值说明硬盘读写太慢拖累 CPU |
| `0.7 hi` | hardware irq | 硬件中断 | 网卡、磁盘等硬件触发 |
| `0.2 si` | software irq | 软件中断 | 系统内部触发 |
| `0.0 st` | steal | 被虚拟机偷走的时间 | 仅在云服务器/虚拟机环境下可见 |

> **判断口诀**：`us + sy` 接近 100% 且 `id` 接近 0% → CPU 被打满；`wa` 高 → 磁盘是瓶颈。

#### 第 4-5 行：内存与 Swap 状态

| 字段 | 含义 | 关键判断 |
|------|------|----------|
| `MiB Mem : 3558.8 total` | 物理内存总量 | — |
| `1416.2 free` | **完全空闲的物理内存** | 不是唯一判断标准 |
| `988.1 used` | 真正被用掉的内存 | — |
| `1397.6 buff/cache` | **缓存内存** | 可自动释放，不算"真正占用" |
| `MiB Swap: 0.0 total` | 交换分区总量 | **为 0 是好事**；用到 Swap 系统会变慢 |

> **内存理解误区**：Linux 会把空闲内存拿来缓存磁盘文件（`buff/cache`），这是**加速机制**，不是内存泄漏。只要 `free` + `buff/cache` 足够，系统就是健康的。

---

### 1.3 下半部分：进程列表（按 CPU 占用排序）

| 列名 | 含义 | 排查重点 |
|------|------|----------|
| `PID` | 进程唯一编号 | 用于后续 `kill` 等操作 |
| `USER` | 启动用户 | `root` 运行且可疑进程 = 高风险 |
| `PR` / `NI` | 进程优先级 | `NI` 负数 = 高优先级，正数 = 低优先级 |
| `VIRT` | 虚拟内存总量（KB） | 申请了但未必实际使用 |
| `RES` | **实际物理内存占用（KB）** | **重点看这个！** |
| `SHR` | 共享内存 | 与其他进程共享的内存大小 |
| `S` | 进程状态 | `R`=运行, `S`=休眠, `D`=不可中断睡眠, `T`=停止, `Z`=僵尸, `I`=空闲 |
| `%CPU` | **CPU 使用率** | **发现问题的核心列** |
| `%MEM` | 内存使用率 | 判断内存大户 |
| `TIME+` | 累计 CPU 时间 | 运行越久、值越高，说明进程越"勤奋" |
| `COMMAND` | 进程名称/命令行 | 识别是什么程序 |

> **状态速查**：`R` 状态过多 → CPU 密集型；`D` 状态过多 → 磁盘 I/O 阻塞；`Z` 状态 → 需要清理。

---

### 1.4 CPU 排查总结：三步法

```
Step 1: 看 load average → 是否超过 CPU 核数？
Step 2: 看 CPU 状态行 → us/sy 谁高？id 是否为 0？wa 是否高？
Step 3: 看进程列表 → 哪个进程 %CPU 高？状态是什么？
```

## 二、内存爆满排查

![[Pasted image 20260705154610.png]]
![[Pasted image 20260705154639.png]]

> 内存问题的排查通常结合 `top`（看进程内存占用）、`free -h`（看整体内存）和 `ps`（精确查找内存大户）三个工具。

### 2.1 常用内存排查命令

```bash
# 查看整体内存使用情况（人类可读格式）
free -h

# 查看进程内存占用排序（类似 top 的进程视图）
ps aux --sort=-%mem | head -n 20

# 查看具体进程的内存细节（替换 <PID>）
cat /proc/<PID>/status | grep -E 'VmRSS|VmSize'

# 查看系统整体内存分配详情
cat /proc/meminfo
```

### 2.2 内存问题常见场景

| 场景 | 现象 | 排查方法 |
|------|------|----------|
| 应用程序内存泄漏 | `RES` 持续增长不释放 | 持续观察 `top` 中该进程的 `RES` 列 |
| 缓存占用过多 | `buff/cache` 很大但 `free` 很小 | 正常现象，必要时用 `echo 3 > /proc/sys/vm/drop_caches` 清理（谨慎） |
| Swap 被大量使用 | 系统极慢，`free` 中 Swap 有值 | 加内存或关闭/减少 Swap 使用 |
| 内存碎片 | 总内存够用但无法分配大块内存 | 查看 `/proc/buddyinfo` |

### 2.3 内存排查总结：三步法

```
Step 1: free -h → 看 total/used/free/available，判断是否真不足
Step 2: top / ps → 定位哪个进程是内存大户
Step 3: 分析进程 → 是正常业务占用还是内存泄漏？是否需要重启或优化？
```

## 三、快速参考卡

| 问题 | 首选命令 | 关键指标 |
|------|----------|----------|
| CPU 过高 | `top` | `load average`, `%CPU`, `us/sy/id/wa` |
| 内存不足 | `free -h` + `top` | `used/total`, `RES`, `%MEM` |
| 磁盘 I/O 瓶颈 | `top` + `iostat` | `wa` 列, `D` 状态进程 |
| 僵尸进程 | `top` + `ps aux \| grep Z` | `zombie` 计数 |

---

> **最后提醒**：性能排查是**先整体后局部**——先判断系统整体状态（上半部分），再定位具体进程（下半部分）。不要一看到某个进程占用高就急着杀，先确认它是不是正常业务进程。
